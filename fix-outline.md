# 大纲问题修复：$ARGUMENTS

**在 check-volume 或 design-conflict-map 发现问题后使用，系统性修复大纲缺陷。**

> **核心理念**：
> - 发现问题不等于解决问题，需要系统性的修复方案
> - 修复要保持大纲整体一致性，不能头痛医头
> - 修复后要验证，确保没有引入新问题

---

### 1.1 读取相关文件

## 第二步：问题分类与诊断

### 2.1 问题类型诊断

**按问题类型分类处理**：

| 问题类型 | 典型表现 | 修复策略 |
|----------|----------|----------|
| **节奏拖沓** | 连续低张力、无冲突间隔过长 | 插入冲突/爆点 |
| **节奏疲劳** | 连续高张力、无喘息 | 插入缓冲章节 |
| **冲突单一** | 只有外部冲突、缺人际/内心 | 叠加多维冲突 |
| **爆点不足** | 爆点密度低、类型单一 | 增加/替换爆点 |
| **弧光缺失** | 无Lie体现、无Truth萌芽 | 增加弧光标注 |
| **伏笔遗漏** | 应回收未回收、应埋设未埋设 | 补充伏笔设计 |
| **衔接断裂** | 与前卷状态不衔接 | 修改开篇承接 |
| **约束违反** | 角色/剧情/时间线约束违反 | 调整剧情设计 |
| **高潮不足** | 高潮章节张力/冲突不够 | 强化高潮设计 |
| **钩子缺失** | 章末/卷末钩子弱 | 重写钩子 |

### 2.2 问题关联分析

**检查问题之间的关联**：

```markdown
## 问题关联分析

### 问题链条识别
- 问题A（节奏拖沓）→ 可能导致 → 问题B（爆点不足）
- 问题C（冲突单一）→ 可能导致 → 问题D（高潮不足）

### 根因分析
| 表层问题 | 可能根因 | 建议 |
|----------|----------|------|
| 第X-X章拖沓 | 缺乏明确的批次目标 | 重新定义批次主题 |
| 高潮章节不够燃 | 缺乏三维冲突叠加 | 增加人际+内心冲突 |
```

## 使用时机

```
/plan-volume X       → 规划大纲
        ↓
/check-volume X      → 检查大纲（发现问题）
        ↓
/design-conflict-map X → 冲突检验（发现更多问题）
        ↓
/fix-outline X       → 修复问题 ← 你在这里
        ↓
/check-volume X      → 重新验证
        ↓
/auto-write X 1      → 开始写作
```

---

## 注意事项

1. **先诊断后修复**：不要看到问题就改，先理解问题的根因
2. **评估影响范围**：修改一章可能影响多章，要考虑连锁反应
3. **保持一致性**：修复要与全书设定、人设、前后文一致
4. **不要过度修复**：轻微问题可以保留，不必追求完美
5. **记录修改**：保留修改记录，便于追溯和回滚
6. **验证修复**：修复后要重新检查，确保没有引入新问题

## 常见修复模式速查

| 问题 | 快速修复 |
